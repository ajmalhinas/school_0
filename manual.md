## 1. Overview

The Androix MDD (Model-Driven Development) framework is designed to simplify Android application development by providing a structured, component-based architecture. It emphasizes separation of concerns between Architecture Engineers and Feature Developers, enabling rapid development of maintainable Android apps.

Key principles:
- **Single Activity Architecture**: Applications have one MainActivity with multiple Fragments.
- **Action-Based Logic**: Business logic is encapsulated in Action classes.
- **Automatic Data Binding**: UI components are automatically bound to model objects via naming conventions.
- **Built-in Persistence**: SQLite-based ORM with transaction management.
- **Validation Framework**: Declarative validation using the Chk utility.

For initial setup and basic usage, refer to [Readme.md](../Readme.md).

## Comparison with Traditional Development

### Traditional Android Development Approach

In traditional Android development without the MDD framework, developers typically:

- **Manual Activity and Fragment Management**: Create multiple Activities and Fragments, handling navigation, back stack, and lifecycle manually.
- **Manual UI Data Binding**: Use findViewById() to get UI components, manually set/get text, handle events with onClick listeners.
- **Scattered Business Logic**: Business logic mixed with UI code in Activities/Fragments, leading to bloated classes.
- **Database Operations**: Use SQLiteOpenHelper, ContentProviders, or Room, writing raw SQL or managing DAOs manually.
- **Threading**: Manually manage threads or AsyncTasks for background operations, handling UI updates on main thread.
- **Validation**: Implement validation logic in UI code, showing errors via Toast or TextView.
- **Error Handling**: Catch exceptions in UI code, handle them locally.

This approach requires more boilerplate code, increases the risk of bugs, and makes maintenance harder as the app grows.

### MDD Framework Approach

The Androix MDD framework provides:

- **Single Activity Architecture**: One MainActivity with multiple Fragments, simplifying navigation and state management.
- **Automatic Data Binding**: UI components bound to model objects via naming conventions, no manual findViewById or setText.
- **Action-Based Logic**: Business logic encapsulated in Action classes, separated from UI.
- **Built-in Persistence**: NPersitence a light weight ORM with automatic transaction management.
- **Asynchronous Operations**: AsyncAction subclasses handle background tasks, UI updates automatically.
- **Declarative Validation**: Chk utility for validations, with automatic error display.
- **Centralized Error Handling**: Framework manages exceptions, displays messages/errors consistently.

### Advantages of MDD Framework

- **Faster Development**: Reduces boilerplate by 60-80%, allowing focus on business logic.
- **Improved Maintainability**: Clear separation of concerns, modular code structure.
- **Consistency**: Standardized patterns ensure uniform code quality.
- **Reduced Errors**: Built-in validation, transaction management, and automatic binding minimize bugs.
- **Scalability**: Easy to add features without affecting existing code.
- **Testability**: Actions can be unit tested independently of UI.
- **Rapid Prototyping**: Quick to build MVPs and iterate on features.

By adopting the MDD framework, teams can deliver Android apps faster, with higher quality and better maintainability compared to traditional approaches.

## 3. Architecture

### 2.1 Application Structure

A typical MDD application follows this structure:
- `app/src/main/java/jas.school/` - Application package
  - `actions/` - Business logic actions
  - `app/` - Application facade and main classes
  - `model/` - Data entities
  - `searchUI/` - Search interfaces
- `app/src/main/assets/school.sql` - Database schema

### 2.2 Framework Layers

1. **Presentation Layer**: Fragments and UI components
2. **Action Layer**: Business logic and data binding
3. **Persistence Layer**: Database operations and transactions
4. **Utility Layer**: Common utilities and helpers

### 2.3 Action Lifecycle

Actions follow a defined lifecycle:
1. `preExecute()` - Setup (e.g., transaction begin)
2. `execute()` - Main business logic
3. `postExecute()` - Teardown (e.g., transaction commit)
4. `updateView()` - UI synchronization
5. Message/Error display

### 2.4 Data Flow within a Action

Data flows from UI → Action → Model → Persistence → UI, with automatic binding at each step.

## 3. Data Models and Entities

### 3.1 Base Classes

All entities must extend [`Identifiable`] or its subclasses:

- `Identifiable`: Basic identifiable entity
- `IdentifiablePrePId`: ID generated before persistence
- `UndeletablePrePId`: Non-deletable entity

Example:
```java
public class Student extends IdentifiablePrePId {
    private String indexNo;
    private String fName;
    private String lName;
    private String address;
    private Integer age;
    // Getters and setters
}
```

### 3.2 Entity Design

- Use wrapper types (Integer, Long) instead of primitives
- BigDecimal for floating-point numbers
- No bidirectional relationships; use findAll() for reverse lookups
- Surrogate keys (long id) for all entities

### 3.3 Relationships

Relationships are unidirectional. For example:
```java
public class Order extends Identifiable {
    private Long customerId; // Reference to Customer
    // No Customer object field
}
```

## 4. Actions

Actions are the core of business logic in MDD framework.

### 4.1 Types of Actions

#### 4.1.1 FBaseAction

Base class for non-persistent operations. See [`FBaseAction`].

Example:
```java
public class CalculateTotalAction extends FBaseAction {
    @Override
    protected void execute() throws Exception {
        BigDecimal amount = getBigDecimal("amount");
        BigDecimal tax = amount.multiply(new BigDecimal("0.1"));
        setBigDecimal("total", amount.add(tax));
    }
}
```

#### 4.1.2 SaveAction

For operations requiring persistence. Extends FBaseAction with transaction management. See [`SaveAction`].

Example:
```java
public class SaveStudentAction extends SaveAction {
    @Override
    protected void execute() throws Exception {
        Student student = (Student) getViewModel("Student");
        F.persist(student);
        addMessage("success", "Student saved successfully");
    }
}
```

#### 4.1.3 AsyncForegroundAction

For long-running operations with UI blocking. See [`AsyncForegroundAction`].

Example:
```java
public class SyncDataAction extends AsyncForegroundAction {
    @Override
    protected void execute() throws Exception {
        // Long operation
        api.syncData();
        executeAfter(() -> {
            addMessage("sync", "Data synchronized");
        });
    }
}
```

#### 4.1.4 AsyncBackgroundAction

For background operations without UI blocking. See [`AsyncBackgroundAction`].

#### 4.1.5 Show Action

Special action automatically invoked when a UI becomes visible. Used to handle data passed from other UIs (e.g., selected items from search).

Example:
```java
public class StudentRegUIShow extends FBaseAction {
    @Override
    public void execute() throws Exception {
        // Handle data passed from search UI
        Student selectedStudent = (Student) F.getFromAppContext("StudentRowSelection");
        if (selectedStudent != null) {
            updateViewModel("Student", selectedStudent);
            F.removeFromAppContext("StudentRowSelection");
        }
    }
}
```

**How it works:**
1. When user selects a student from search list, framework stores it in app context with key `"StudentRowSelection"`
2. Framework navigates to `StudentRegUI` and calls `onShow()` method
3. `onShow()` automatically invokes `StudentRegUIShow` action (naming convention: `UIClassName + "show" + Action`)
4. Action retrieves selected data and populates form fields
5. User sees complete student details ready for viewing/editing

#### 4.1.6 PeriodicAction

For recurring tasks. Not shown in current codebase but referenced in manual.

### 5.3.1 SQLBuilder

The SQLBuilder class provides a fluent API for constructing SQL statements programmatically, offering an alternative to raw SQL string manipulation within the MDD framework. It is used to create non parmeterized SQL statements. 

#### Basic Usage

##### SELECT Statements

```java
// Simple SELECT all fields
SQLBuilder query = SQLBuilder.select("student", "*");

// SELECT specific fields
SQLBuilder query = SQLBuilder.select("student", "id, name, age");

// SELECT with WHERE condition
SQLBuilder query = SQLBuilder.select("student", "id, name, age")
    .where("age").gt(18);

// SELECT with JOIN
SQLBuilder query = SQLBuilder.select("student s", "s.name, c.course_name")
    .join("course", "c", "s.course_id = c.id");

// SELECT with multiple WHERE conditions
SQLBuilder query = SQLBuilder.select("student", "*")
    .where("age").gt(18)
    .and("fName").like("John%");

// SELECT with ORDER BY and LIMIT
SQLBuilder query = SQLBuilder.select("student", "*")
    .where("age").gt(18)
    .orderBy("name", SQLBuilder.ASC)
    .limit(10);

// SELECT with GROUP BY
SQLBuilder query = SQLBuilder.select("student", "department, COUNT(*)")
    .groupBy("department");

// Get the SQL string
String sql = query.getStatement();
```

##### INSERT Statements

```java
// INSERT with values
SQLBuilder insert = SQLBuilder.insert("student")
    .set("name", "John Doe")
    .set("age", 20)
    .set("email", "john@example.com");

String sql = insert.getStatement();
```

#### Advanced Examples

##### Complex SELECT with Multiple Joins and Conditions

```java
SQLBuilder complexQuery = SQLBuilder.select("student s", "s.name, c.course_name, d.department_name")
    .join("course", "c", "s.course_id = c.id")
    .join("department", "d", "c.department_id = d.id")
    .where("s.age").gt(18)
    .and("c.credits").gt(3)
    .or("d.name").eq("Computer Science")
    .orderBy("s.name", SQLBuilder.ASC)
    .orderBy("c.course_name", SQLBuilder.DESC)
    .limit(50);

String sql = complexQuery.getStatement();
```

#### Advantages over Other Approaches

Compared to raw SQL string concatenation:
- Less prone to syntax errors and SQL injection vulnerabilities
- Easier to modify and extend queries
- Better readability and maintainability
- Automatic handling of SQL escaping and formatting

Compared to other query builders like Psql:
- More lightweight with minimal abstraction overhead
- Direct SQL generation without additional parsing layers
- Tighter integration with the MDD framework's persistence layer
- Simpler API for basic CRUD operations

The SQLBuilder is particularly useful when you need fine-grained control over SQL generation while maintaining type safety and preventing common SQL construction errors.

## 8. Search and Querying

### 8.1 SearchUIs

Search UIs in the MDD framework provide a standardized way to display and search through collections of model objects. They follow a Model-View-Controller pattern where:

- **Model**: Your entity classes (e.g., Student)
- **View**: Row layouts and search UI layout
- **Controller**: Row classes that handle data binding and search filtering

### 8.1.1 Search UI Architecture

The search UI consists of several components working together:

1. **Search UI Class** (`ModelSearchUI.java`): Manages the overall search interface and data loading
2. **Search UI's Layout** (`search_ui.xml`): Contains a RecyclerView that displays the list of items
3. **Row Class** (`{Model}Row.java`): Handles data binding and search filtering logic
2. **Row's Layout** (`{model}_row.xml`): Defines how each individual item appears in the list

**Data Flow:**
```
Database → ModelSearchUI.setup() → Row Class → RecyclerView → Row Layout
Search Input → Row.match() → Filtered Results → UI Update
```




### 8.1.2 Generic Steps to Create a Default Search UI for a Model

1. **Create the Row Class**:

   Create a new Java class in the `rows` package (e.g., `{ModelName}Row.java`) that extends `AbstractSearchUIRow`.
   Implement the required methods: `makeRowView()` to initialize views, `bindRowView()` to bind data, and `match()` for search filtering.

   ```java
   package jas.school.rows;

   import android.view.View;
   import android.widget.TextView;

   import com.androix.AbstractSearchUIRow;

   import java.util.List;

   import jas.school.app.R;
   import jas.school.model.Student;

   
   public class StudentRow extends AbstractSearchUIRow {
       private TextView indexNoTextView, nameTextView;
       public StudentRow(View v) {
           super(v);
       }

       @Override
       public void makeRowView(View v) {
           indexNoTextView = v.findViewById(R.id.studentRowIndexNo);
           nameTextView = v.findViewById(R.id.studentRowName);
       }

       @Override
       public AbstractSearchUIRow bindRowView(AbstractSearchUIRow row, int position, List listItems) {
           StudentRow sr = (StudentRow) row;
           Student student = (Student) listItems.get(position);
           sr.indexNoTextView.setText(student.getIndexNo());
           String fullName = (student.getFName() != null ? student.getFName() : "") + " " +
                             (student.getLName() != null ? student.getLName() : "");
           sr.nameTextView.setText(fullName.trim());
           return sr;
       }

       @Override
       public boolean match(Object o, String searchTerm) {
           if (searchTerm == null || searchTerm.trim().isEmpty()) return true;
           Student student = (Student) o;
           String term = searchTerm.toLowerCase();
           return (student.getIndexNo() != null && student.getIndexNo().toLowerCase().contains(term)) ||
                  (student.getFName() != null && student.getFName().toLowerCase().contains(term)) ||
                  (student.getLName() != null && student.getLName().toLowerCase().contains(term));
       }
   }
   ```

2. **Create the Layout XML for the Row**:

   Create a new XML file `{model_name}_row.xml` (e.g., `student_row.xml`).
   Define a root view (e.g., LinearLayout or CardView) with an ID for the row card.

### 8.1.3 Discussion
   Registration of the SearchUI must be done before the navigation. However, it is not mandotory to do that in the
   Registration class.
- **Why Register in Registration.java?** This is a good practice because:
  - **Centralization**: All registrations are in one place (`registerSearchUIs()`), making it easier to manage, update, and avoid duplicates. The framework calls this method during app initialization (via `AbstractRegistration`), ensuring registrations happen early and only once.
  - **Reusability**: Once registered, the search UI can be navigated to from multiple places (e.g., menus, buttons) without re-registering, reducing code duplication.
  - **Separation of Concerns**: Registration defines the UI structure (layouts, classes), while navigation handles user flow.
  - **Consistency**.

## 11. Best Practices

- Follow naming conventions strictly
- Use SaveAction for persistence operations
- Validate early in actions
- Avoid manual thread creation; use AsyncActions
- Enforce database constraints at schema level
- Use TDRE for requirements (see [Readme.md](../Readme.md) section 13)
- Separate concerns: Architecture Engineers vs Feature Developers

## 12. Troubleshooting

### 12.1 Common Errors

- **"Attempt to invoke virtual method on null object"**: Missing entity fields for DB columns. Ensure all table columns mapped.
- **"NPersistence not initialized"**: Call NPersistence.setup() first.
- **Transaction active**: Ensure proper commit/rollback.

### 12.2 Limitations

- Actions update values only, not UI properties
- Single connection SQLite model

### 12.3 Debugging

- Use Android Studio Database Inspector
- Check logs with TAG_STAT
- Validate naming conventions
